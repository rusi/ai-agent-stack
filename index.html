Step 1 · The Foundation

The Model (LLM)

A large language model — or just "model" — is software trained on enormous amounts of text. You send it a message, it predicts the most likely response, token by token.

No internet access. No memory between conversations. No tools. Just text in, text out.

Example"What is the capital of France?" → "Paris." The model knows this from training data — it isn't looking anything up.
Step 2 · Structure

System Prompt & Structured Output

Before your message, the model receives a system prompt — hidden instructions that shape its behavior. Together, this forms the context window.

If the system prompt says "reply in JSON," the model outputs structured data instead of prose. Now software can parse it — and that changes everything.

Why this mattersStructured output is what makes tools, MCP, and agents possible. It's how a model "talks" to code instead of to humans.
Step 3 · Action

Tool Use (Function Calling)

Now the model can do things it wasn't trained on. Ask "What's the weather in Tokyo?" and it outputs a structured tool request — like get_weather("Tokyo").

A separate piece of code — the Tool Executor — runs the actual function, gets real-time data, and sends the result back. The model then writes the final answer.

Key insightThe model only outputs a JSON request. Separate code does the work. This is a crucial security boundary — the model never executes anything directly.
Step 4 · Connectivity

MCP — Model Context Protocol

MCP is a universal standard for connecting AI to external services — think USB-C for AI tools. Build one connector, any compatible model can use it.

The available MCP tools are listed inside the system prompt — that's how the model knows what it can ask for. The MCP Server handles the actual API calls.

One protocol, hundreds of integrationsGmail, Slack, GitHub, Salesforce, Jira, PostgreSQL, Stripe — build once, connect everywhere.
Step 5 · Power

Bash & Terminal Access

The executor can also run shell commands — install packages, read files, execute scripts, run tests. It's just another tool type, same pattern.

Model outputs bash("npm test"), executor runs it in a shell, result comes back. Simple — but powerful.

This is where risk increasesThe AI can now execute code on your machine. Sandboxing, permission boundaries, and approval workflows become critical.
Step 6 · Identity

AGENTS.md — The Persona

A markdown file injected into the system prompt at startup. It defines who the agent is: its role, rules, allowed tools, and constraints.

Different AGENTS.md = completely different agent. Same model, totally different behavior and expertise.

In practiceClaude Code uses CLAUDE.md, Cursor uses .cursorrules, Codex uses custom instructions. Different names, same idea.
Step 7 · Expertise

SKILL.md — Task Expertise

Skill files are loaded on-demand when the agent detects a relevant task. They contain step-by-step procedures, templates, and quality checks for specific workflows.

AGENTS.md = who you are (always loaded). SKILL.md = how to do a specific task (loaded when needed).

ExamplesCreating slide decks, generating PDFs, reviewing code, writing tests — each skill file encodes domain best practices the model wouldn't know otherwise.
Step 8 · Autonomy

The Agentic Loop

This is the key ingredient. Instead of one request → one response, the system keeps going. It plans what to do, picks a tool, executes, observes the result, and decides what's next — all on its own.

A single user message might trigger dozens of tool calls, file reads, and test runs — all autonomously, in a loop. This is what turns a model + tools into an agent.

Agent = Model + Tools + Instructions + LoopThe model predicts tokens. The tools take actions. The instructions define behavior. The loop gives it agency — the ability to decide what happens next.
Step 9 · Scale

Multi-Agent Orchestration

An orchestrator spawns multiple copies of this entire stack — each with different personas, skills, and tool access. One agent researches, one codes, one reviews.

They pass results between each other like a virtual engineering team. The orchestrator decides who does what and when.

This is the full pictureClaude Code's background agents, Cursor's parallel editors, and Codex's task runners all use this multi-agent pattern.

Real World

Same Stack, Different Products

Every AI coding tool is built from the same blocks. What makes each one an agent is the same two things: a tool executor (it can take actions) and injected instructions (it has a persona and rules). The model alone is just a prediction engine.

Claude Code Anthropic · CLI

Model: Claude (Anthropic API)
Tools: Bash, file system, MCP servers
Instructions: CLAUDE.md injected into system prompt
Skills: SKILL.md loaded on-demand per task
Multi-agent: Background subagents with isolated contexts

Cursor Anysphere · IDE

Model: Multi-LLM (Claude, GPT, custom)
Tools: Terminal, codebase indexing, visual editor
Instructions: .cursorrules injected into system prompt
Skills: Built-in workflows (edit, compose, review)
Multi-agent: Parallel background agents per file

Codex OpenAI · Cloud

Model: GPT-4o / o3 (OpenAI API)
Tools: Sandboxed cloud environment, GitHub
Instructions: Custom instructions in project config
Skills: N/A (single workflow)
Multi-agent: Parallel cloud tasks per issue

OpenCode Open Source · CLI

Model: Any LLM — or another agent (see below)
Tools: Bash, file system, MCP servers
Instructions: AGENTS.md injected into system prompt
Skills: Community-contributed skill files
Multi-agent: Configurable sub-processes
The agent-as-model pattern OpenCode can use Claude CLI as its model — even though Claude CLI is itself an agent with its own tools and instructions. From OpenCode's perspective, it's just an API that accepts text and returns text. But under the hood, that "model" is a full agent stack: Claude (the LLM) + CLAUDE.md (instructions) + Bash + MCP tools. This is how agent stacks compose — each layer can wrap the one below it.
What makes it an agent? Not the model. The model just predicts tokens. An agent = model + tool executor + instructions + loop.
Governance

5 Questions Before Approving Any AI Tool

A framework for IT, Legal, and leadership teams evaluating AI agent products.

1

Where does data go?

Local processing, cloud API, or enterprise VPC? Who sees the prompts and outputs?

2

What can it do?

Read-only analysis? Or can it modify files, send emails, execute code, deploy?

3

What's connected?

Which MCP tools and integrations? What permissions do they have?

4

Who controls it?

Admin oversight, audit logging, approval workflows, kill switch?

5

What could go wrong?

Worst-case scenario? Can actions be reversed? What's the blast radius?
