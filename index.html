The AI Agent Stack

From Models to Multi-Agent Systems

An AI agent is not a new kind of model. It is a model API call wrapped with instructions, context, tools, permissions, and a loop that keeps working until the task is done.

Step 1 · The Core

The Model (LLM)

A Large Language Model is the engine. You send it a prompt 1 through an API, and it predicts the next tokens in its reply 2.

On its own, this is just a stateless inference call: text in, text out. No durable memory, no file edits, and no ability to act in the world by itself.

Stateless EngineA plain model call 1 can answer, but it cannot inspect your repo or run tools unless something around it is built to do that.
Step 2 · Structure

Context & Prompts

Before the model sees your message, the agent wrapper assembles a context window 3 with system instructions, chat history, and any structured state it wants the model to consider.

This is where the same base model becomes an assistant, a reviewer, or a planner. It is also how products ask the model to emit JSON or tool calls instead of free-form chat.

InitializationThe wrapper decides what goes into context 3. That framing is often as important as the model itself.
Step 3 · Persona

Instruction Files

Many agent products load project-level instruction files 4, such as AGENTS.md, CLAUDE.md, or .cursor/rules, into the model context.

These files define role, standards, workflow expectations, and guardrails. Swapping the instruction layer can turn the same model into a security reviewer, staff engineer, or product writer.

Persistent RulesInstruction files 4 are a reusable policy layer on top of the raw model API.
Step 4 · Expertise

Reusable Workflows

Some systems also load specialized workflow modules 5, like skills, playbooks, or subagent prompts, only when they are relevant to the task.

This keeps the base context lean while still giving the agent structured expertise for things like PR review, deploys, migrations, or debugging.

Just-in-TimeTask-specific expertise 5 is loaded on demand instead of bloating every prompt.
Step 5 · Action

The Tool Executor

The model emits a tool request 6. A real executor process in the agent wrapper runs the tool, gathers the output, and sends the result back into context 7.

The model then interprets that tool result and writes the final answer 8. This model-plus-executor boundary is where a chat UI becomes an agent runtime.

Execution BoundaryThe model decides what to do. The executor 7 is the code that actually does it.
Step 6 · Connectivity

MCP · Universal Plugins

Model Context Protocol (MCP) 9 standardizes how agents discover and call tools exposed by external servers.

Not every integration is MCP, but MCP makes connectors more portable across products by giving tools a shared interface instead of one-off custom glue.

InteroperabilityMCP 9 sits between the agent runtime and outside services, not inside the model itself.
Step 7 · Direct Power

Bash & System Access

The highest-leverage tools are often local ones: shell commands, file writes, tests, browsers, package managers, and deployment scripts 10.

That is also why permissions matter so much. Sandboxes, approvals, network policies, and audit logs are part of the architecture, not optional afterthoughts.

PermissionsThe permission layer 10 defines the blast radius of the agent.
Step 8 · Autonomy

The Autonomous Loop

This is the actual jump from assistant to agent. Instead of stopping after one answer, the system loops 11: inspect state, choose the next action, run it, evaluate the result, and continue.

AgencyThe loop 11 is what lets the system persist across a multi-step task.
Step 9 · Swarms

Orchestration at Scale

Finally, an orchestrator 12 can spawn multiple agents with separate contexts, tool scopes, or sandboxes. They delegate work, report back, and sometimes run in parallel.

DelegationMulti-agent systems 12 are composition: many agent runtimes coordinated together.

Same Blocks, Different Products

Every modern AI coding tool uses the same building blocks. What changes is the model layer, the instruction system, and where execution actually happens: in your editor, on your machine, or in a remote sandbox.

Claude Code Anthropic · CLI

Model: Claude models
Instructions: CLAUDE.md
Runtime: Local terminal agent with subagents and tool permissions.
Key Strength: Strong terminal workflow with explicit memory and delegation primitives.

Codex OpenAI · CLI / Cloud

Model: GPT-5.4
Instructions: AGENTS.md and runtime policy
Runtime: Local CLI or cloud task sandboxes, depending on surface.
Key Strength: Strong coding and agentic execution across tools and computer-use workflows.

Cursor Anysphere · IDE

Model: Multi-model selection
Instructions: .cursor/rules
Runtime: In-editor agent plus optional remote background agents.
Key Strength: Deep IDE integration, codebase indexing, and async agents.

Gemini Code Assist Google · IDE / CLI

Model: Gemini family
Instructions: Workspace context and agent-mode prompts
Runtime: IDE assistant plus Gemini CLI agent loop.
Key Strength: Tight Google ecosystem integration and cited code assistance.

OpenCode Open Source · Terminal / App

Model: Any configured provider API or local model
Instructions: Config, prompts, and selected tools
Runtime: Its own agent wrapper; not just a thin skin over another coding agent CLI.
Key Strength: Provider-agnostic runtime with direct model selection and local execution.

OpenClaw Open Source · Gateway

Model: Provider configurable
Instructions: AGENTS.md, SOUL.md, TOOLS.md, and skills
Runtime: A self-hosted gateway that owns channels, sessions, agents, and optional remote nodes.
Key Strength: Strong control plane for multi-agent routing, approvals, and execution across devices.
Direct Model Path OpenCode makes the distinction explicit: you connect provider APIs, choose a model, and its local runtime wraps that model with prompts, tools, and execution logic. The agent is the wrapper, not the raw model.
Gateway + Nodes Path OpenClaw adds a heavier control plane: a long-lived gateway owns sessions, channels, and agent routing, then forwards execution to local or remote node hosts behind approvals and per-agent policies.

Evaluating the Stack

A simple checklist for IT and Leadership teams before adopting agentic tools.

1

Data Sovereignty

Where is the model running, and who stores the logs of my prompt history?

2

Tool Permissions

What is the 'blast radius' if the agent goes rogue? Does it have write access?

3

Auditability

Can I replay every action the agent took in a verifiable log?

4

Approval Gates

Where are the 'human-in-the-loop' checkpoints for high-risk actions?

5

Instruction Drift

How do I prevent the agent from deviating from its core AGENTS.md policy?
