Step 1 · The Core

The Model (LLM)

A Large Language Model — the engine of the stack. You send it a message 1, and it predicts the most likely response 2.

At this stage, it's just text-in, text-out. No internet access, no memory, and no ability to take action.

Stateless EngineThe model alone 1 has no persistence. Every turn is a brand new start.
Step 2 · Structure

Context & Prompts

Before the model sees your message, it is initialized with a Context Window 3 containing system prompts and history.

This hidden context defines the "rules of the game." By instructing the model to "reply in JSON," you turn a chat interface into a machine-readable API.

InitializationInstructions 3 are injected first, shaping everything that follows.
Step 3 · Persona

AGENTS.md · Identity

The AGENTS.md file 4 is injected into the context to define the agent's core identity: its role, rules, and behavioral boundaries.

By swapping this file, you can turn the same model into a security auditor, a codebase architect, or a creative copywriter.

Persistent RulesThe Identity 4 provides the moral and professional compass for the model.
Step 4 · Expertise

SKILL.md · Workflows

Specialized Skill Files 5 load on demand. They contain "recipes" for complex tasks like "AWS Deployment" or "PR Review."

This keeps the context lean by only loading specific expertise when a trigger keyword is matched in your intent.

Just-in-TimeExpertise 5 is added only when relevant, then discarded.
Step 5 · Action

The Tool Executor

The model outputs a tool request 6. The Tool Executor — real code running in your agent wrapper — performs the work and feeds the result back 7.

The model then writes the final synthesized response 8 back to the user.

Execution LoopResults 7 flow back to the brain (model) for final analysis.
Step 6 · Connectivity

MCP · Universal Plugins

Model Context Protocol (MCP) 9 is the universal standard for AI tools. It allows any agent to talk to any external service like GitHub, Jira, or your custom DB.

InteroperabilityStandardized calls 9 mean tools work across different agent products.
Step 7 · Direct Power

Bash & System Access

The agent can directly execute shell commands 10. This is where governance becomes part of the architecture—sandboxing and approval gates ensure the agent remains safe.

PermissionsThe policy 10 determines the blast radius of the agent's actions.
Step 8 · Autonomy

The Autonomous Loop

This is the soul of an agent. Instead of stopping, the system loops 11: it observes its own work, corrects errors, and persists until the task is complete.

AgencyThe loop 11 transforms a chatbot into a proactive coworker.
Step 9 · Swarms

Orchestration at Scale

Finally, a Multi-Agent Orchestrator 12 spawns multiple instances of this stack. They delegate sub-tasks to each other, acting as a virtual engineering department.

DelegationComplex goals are achieved via parallel specialized agents 12.

Same Blocks, Different Products

Every modern AI tool uses these building blocks. The difference lies in how they combine them and where the execution boundaries are drawn.

Instructions: CLAUDE.md

Model: Multi-model selection
Instructions: .cursor/rules
Key Strength: Native IDE integration and codebase indexing.

Key Strength: Massive context window (up to 2M tokens).

Evaluating the Stack

A simple checklist for IT and Leadership teams before adopting agentic tools.

1

Data Sovereignty

Where is the model running, and who stores the logs of my prompt history?

2

Tool Permissions

What is the 'blast radius' if the agent goes rogue? Does it have write access?

3

Auditability

Can I replay every action the agent took in a verifiable log?

4

Approval Gates

Where are the 'human-in-the-loop' checkpoints for high-risk actions?

5

Instruction Drift

How do I prevent the agent from deviating from its core AGENTS.md policy?
