The AI Agent Stack

From Models to Multi-Agent Systems

An AI agent is not a new kind of model. It is a model API call wrapped with instructions, context, tools, permissions, and a loop that keeps working until the task is done.

Step 1 · The Core

The Model (LLM)

A Large Language Model is an external API — a remote inference service. On its own it is stateless: no memory, no file access, no ability to act.

The model sits on the right. Your code calls it. Everything else you see in this guide is the wrapper around that call.

Step 1 · The Prompt

You Send a Prompt

You send text to the model API 1. The model receives your question as a sequence of tokens — nothing more.

Step 1 · The Response

It Predicts a Reply

The model predicts the most likely next tokens 2. Text in, text out. No tools, no memory, no side effects.

StatelessEach API call is independent. The model knows facts, reasoning, and code patterns from training — but it cannot look things up or take actions.
Step 2 · The Runtime

The Agent Runtime

Between you and the model sits the agent runtime — a local wrapper that assembles context, calls the model API, and interprets what comes back.

At its core is the Execution Engine: the component that receives your request and coordinates everything that follows.

Step 2 · The Request

Your Question Enters the Runtime

You ask: “What is the weather in Boston?” 1. This time the question goes to the Execution Engine — not directly to the model.

Step 2 · Context

Context Window

The runtime assembles a context window — system prompt, conversation history, and configuration — then forwards the question to the model 2.

The context window is the lens through which the model sees everything. Change the context, change the behavior.

FramingThe same base model becomes an assistant, a reviewer, or a planner — depending on what the runtime puts in the context window.
Step 3 · Tool Use

Tool Calls & Results

The model cannot check live weather — but it can ask for help. It returns a tool call 3 instead of a plain answer.

The Execution Engine runs the tool locally, collects the result 4, feeds it back into context, and calls the model again. The model now has real data and produces the final answer 5.

Execution BoundaryThe model decides what to do. The engine actually does it — locally, on your machine — then feeds the result back.
Step 4 · Connectivity

MCP · Universal Plugins

Model Context Protocol (MCP) standardizes how agents discover and call tools. MCP servers expose tool schemas 5 — the runtime injects these into the context assembly so the model knows what tools are available.

When the model emits a tool call, the executor routes it to the right MCP server 6. This makes connectors portable across different agent products.

How models discover toolsTool schemas go into the context — that is how the model knows what it can ask for. The MCP server handles the actual execution.
Step 5 · Persona

Instruction Files

On the user side, project-level instruction files 7 get loaded into the runtime's context assembly. Claude Code uses CLAUDE.md. OpenAI Codex uses AGENTS.md. Cursor uses .cursor/rules.

These files define role, standards, workflow expectations, and guardrails. Swapping the instruction layer turns the same model into a security reviewer, staff engineer, or product writer.

Your Rules, Your FilesInstruction files 7 live on your machine, not in the model. The runtime injects them into every prompt.
Step 6 · Expertise

Reusable Workflows

A skill registry 8 lists available workflows in the context — names and short descriptions the model can read.

When a skill is relevant, the model makes a tool call to load it. The executor reads the skill file and returns its content as a tool result, injecting it into context. "Load on demand" is just another tool call — the skill content comes back like any tool result.

On-Demand LoadingThe registry 8 lives in the prompt. Loading a skill is a tool call that returns the skill's full content into context.
Step 7 · Direct Power

Bash & System Access

The highest-leverage tools are local ones: shell commands, file writes, test runners, package managers 9. These run inside the agent runtime, giving the model access to your entire development environment.

A prompt like "create a minesweeper game" can write files, install packages, and run tests — all through the executor.

Local PowerBash 9 is just another tool the executor can call — but it gives the agent access to your entire filesystem and toolchain.
Step 8 · Autonomy

The Autonomous Loop

This is the jump from assistant to agent. The model itself has not changed — it still handles one prompt and one prediction at a time.

The loop 10 comes from instructions in the context. The persona tells the model: plan your approach, take an action, observe the result, decide if you are done. The runtime wrapper re-invokes the model with updated context after each step — wrapping around both the runtime and the model API.

Instructions, Not MagicThe model does not "loop" — the wrapper does. Plan 10 → act → observe → repeat, until the task is complete.
Step 9 · Swarms

Orchestration at Scale

An orchestrator 11 can spawn multiple agent runtimes — each with its own context, tool scope, and model connection. They delegate work, report back, and sometimes run in parallel.

DelegationMulti-agent systems 11 are composition: many runtimes coordinated, each calling its own model API independently.

Implementation

Same Blocks, Different Products

Every modern AI coding tool uses the same building blocks. What changes is the model layer, the instruction system, and where execution actually happens: in your editor, on your machine, or in a remote sandbox.

Claude Code Anthropic · CLI

Model: Claude models
Instructions: CLAUDE.md
Runtime: Local terminal agent with subagents and tool permissions.
Key Strength: Strong terminal workflow with explicit memory and delegation primitives.

Codex OpenAI · CLI / Cloud

Model: GPT-5.4
Instructions: AGENTS.md and runtime policy
Runtime: Local CLI or cloud task sandboxes, depending on surface.
Key Strength: Strong coding and agentic execution across tools and computer-use workflows.

Cursor Anysphere · IDE

Model: Multi-model selection
Instructions: .cursor/rules
Runtime: In-editor agent plus optional remote background agents.
Key Strength: Deep IDE integration, codebase indexing, and async agents.

Gemini Code Assist Google · IDE / CLI

Model: Gemini family
Instructions: Workspace context and agent-mode prompts
Runtime: IDE assistant plus Gemini CLI agent loop.
Key Strength: Tight Google ecosystem integration and cited code assistance.

OpenCode Open Source · Terminal / App

Model: Any configured provider API or local model
Instructions: Config, prompts, and selected tools
Runtime: Its own agent wrapper; not just a thin skin over another coding agent CLI.
Key Strength: Provider-agnostic runtime with direct model selection and local execution.

OpenClaw Open Source · Gateway

Model: Provider configurable
Instructions: AGENTS.md, SOUL.md, TOOLS.md, and skills
Runtime: A self-hosted gateway that owns channels, sessions, agents, and optional remote nodes.
Key Strength: Strong control plane for multi-agent routing, approvals, and execution across devices.
Direct Model Path OpenCode makes the distinction explicit: you connect provider APIs, choose a model, and its local runtime wraps that model with prompts, tools, and execution logic. The agent is the wrapper, not the raw model.
Gateway + Nodes Path OpenClaw adds a heavier control plane: a long-lived gateway owns sessions, channels, and agent routing, then forwards execution to local or remote node hosts behind approvals and per-agent policies.
Governance

Evaluating the Stack

A simple checklist for IT and Leadership teams before adopting agentic tools.

1

Data Sovereignty

Where is the model running, and who stores the logs of my prompt history?

2

Tool Permissions

What is the 'blast radius' if the agent goes rogue? Does it have write access?

3

Auditability

Can I replay every action the agent took in a verifiable log?

4

Approval Gates

Where are the 'human-in-the-loop' checkpoints for high-risk actions?

5

Instruction Drift

How do I prevent the agent from deviating from its core AGENTS.md policy?
