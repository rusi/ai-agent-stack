Step 1 · The Foundation

The Model (LLM)

A large language model — or just "model" — is software trained on enormous amounts of text. You send it a message, it predicts the most likely response, token by token.

No internet access. No memory between conversations. No tools. Just text in, text out. Many products stop here and simply make a plain API call to a model provider.

Example"What is the capital of France?" → "Paris." The model knows this from training data — it isn't looking anything up.
Step 2 · Structure

System Prompt & Structured Output

Before your message, the model receives a system prompt — hidden instructions that shape its behavior. Along with your message, prior turns, tool definitions, and any injected files, this makes up the context window: everything the model can "see" for this conversation.

If the system prompt says "reply in JSON," the model outputs structured data instead of prose. Now software can parse it — and that changes everything.

Why this mattersStructured output is what makes tools, MCP, and agents possible. It's how a model "talks" to code instead of to humans.
Step 3 · Identity

AGENTS.md — The Persona

A markdown file injected into the system prompt when the agent starts. It defines who the agent is: its role, rules, allowed tools, and constraints.

Different AGENTS.md = completely different agent. Same model, totally different behavior and expertise.

In practiceDifferent products package this differently: Claude Code uses CLAUDE.md, Cursor uses .cursor/rules or AGENTS.md, and OpenAI Codex can follow AGENTS.md files in your repo. Different names, same idea — instructions injected into model context before acting.
Step 4 · Expertise

SKILL.md — On-Demand Expertise

Unlike AGENTS.md (always loaded), skill files load on demand. They contain step-by-step procedures, templates, and quality checks for specific workflows.

The agent keeps a skill registry — a list of trigger descriptions like "use when committing code" or "use when creating PDFs." When your input matches a trigger, the skill content is injected into the context window. No match? No skill loads. This keeps the context lean.

How it worksYou type /commit → the agent matches the "commit" trigger → loads procedures for staging, diff review, and message formatting. That expertise wasn't in the context before — it's injected just in time, then unloaded when done.
Step 5 · Action

Tool Use & the Agent Boundary

Now the Agent Application enters the picture — a wrapper program that sits between you and the model. It assembles the context, calls the model provider API, and handles the responses.

When you ask "What's the weather in Tokyo?", the model outputs a structured tool request. The agent's Tool Executor — separate code, not the model — runs the function, gets real-time data, and feeds the result back. The model then writes the final answer.

The five boundariesYou (user) → Agent Application (local wrapper) → Model Provider API → Model. Plus: instructions feed in, and MCP servers connect out. The model decides what to call; the agent decides how to execute it.
Step 6 · Connectivity

MCP — Model Context Protocol

MCP is a universal standard for connecting AI to external services — think USB-C for AI tools. Build one connector, any compatible agent can use it.

The available MCP tools are listed in the model context — that's how the model knows what it can ask for. The MCP Server handles the actual API calls when the executor invokes it.

One protocol, hundreds of integrationsGmail, Slack, GitHub, Salesforce, Jira, PostgreSQL, Stripe — build once, connect everywhere.
Step 7 · Power

Bash, Permissions & Terminal Access

The executor can also run shell commands — install packages, read files, execute scripts, run tests. It's just another tool type, same pattern.

But this is also where policy becomes part of the architecture. Before a command runs, good agents check sandbox rules, allowlists, approval gates, and environment boundaries.

This is where risk increasesThe model can suggest an action. The executor still has to decide whether that action is allowed. Sandboxing, permission boundaries, and approval workflows are not optional extras — they are part of the agent stack.
Step 8 · Autonomy

The Agentic Loop

This is the key ingredient. Instead of one request → one response, the system keeps going. It plans what to do, picks a tool, executes, observes the result, and decides what's next — all on its own.

A single user message might trigger dozens of tool calls, file reads, and test runs — all autonomously, in a loop. This is what turns a model + tools into an agent.

Agent = Model + Tools + Instructions + LoopThe model predicts tokens. The tools take actions. The instructions define behavior. The loop gives it agency — the ability to decide what happens next.
Step 9 · Scale

Multi-Agent Orchestration

An orchestrator spawns multiple copies of this entire stack — each with different personas, skills, and tool access. One agent researches, one codes, one reviews.

They pass results between each other like a virtual engineering team. The orchestrator decides who does what and when.

This is the full pictureClaude Code's background agents, Cursor's parallel editors, and Codex's task runners all use this multi-agent pattern.

Real World

Same Stack, Different Products

Every AI coding tool is built from the same blocks. The important distinction is upstream architecture: some products mostly call model APIs directly; others also orchestrate other agents or remote sandboxes. The model alone is still just a prediction engine.

Claude Code Anthropic · CLI

Model: Claude (Anthropic API)
Tools: Bash, file system, MCP servers
Instructions: CLAUDE.md project and user memory files
Specialization: Custom slash commands and subagents
Multi-agent: Subagents with separate contexts and tool scopes

Cursor Anysphere · IDE

Model: Multi-LLM (Claude, GPT, custom)
Tools: Terminal, codebase indexing, visual editor
Instructions: .cursor/rules or AGENTS.md (.cursorrules is legacy)
Specialization: Scoped project rules, user rules, and memories
Multi-agent: Background agents in isolated remote environments

Codex OpenAI · Cloud

Model: GPT-5.4
Tools: Sandboxed cloud container, tests, linters, GitHub workflows
Instructions: AGENTS.md files in the repo can guide execution
Specialization: Environment config and task-specific delegation
Multi-agent: Parallel cloud tasks running in separate sandboxes

OpenCode Open Source · CLI

Model path: Usually direct provider API calls
Providers: OpenAI, Anthropic, Gemini, local models, and more
Tools: Bash, file system, MCP servers
Instructions: AGENTS.md and compatible project guidance files
Specialization: Native SKILL.md support via the skill tool
Multi-agent: Primary agents plus configurable subagents
Pattern A: direct model API Agent app → provider API → model. This is the common case for tools like OpenCode.
Pattern B: agent-to-agent delegation Agent app → another agent or remote sandbox → model + tools. That is no longer a raw model call; it is a higher-level execution boundary.
What makes it an agent? Not the brand name and not the model alone. An agent = model + tool executor + instructions + permissions + loop.
Governance

5 Questions Before Approving Any AI Tool

A framework for IT, Legal, and leadership teams evaluating AI agent products.

1

Where does data go?

Local processing, cloud API, or enterprise VPC? Who sees the prompts and outputs?

2

What can it do?

Read-only analysis? Or can it modify files, send emails, execute code, deploy?

3

What's connected?

Which MCP tools and integrations? What permissions do they have?

4

Who controls it?

Admin oversight, audit logging, approval workflows, kill switch?

5

What could go wrong?

Worst-case scenario? Can actions be reversed? What's the blast radius?

Key Takeaways

Same blocks everywhereModel + structured output + tools + loop.
Instructions shape behaviorAGENTS.md = who. SKILL.md = workflow expertise.
MCP is the connectorOne standard for all external services.
Permissions are architectureSandboxing and approvals decide what the agent may actually do.
Loops give agencyPlan → Act → Observe → Repeat = agent.
Deployment mattersKnow where your data flows.
Direct API is not delegationCalling a model provider is different from handing work to another agent.
Don't block AI — build a framework to evaluate and govern it.
